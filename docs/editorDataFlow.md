## 核心原则
云端新帖子的id分配必须保持一致性，权威性

## 本地模式
不分配id，直接读写文件

## 上传流程
|顺序|后台行为|前台交互
|---|---|---|
|1|先检查是否已经登录，服务是否可用|静默
|2|调用api，为待上传的帖子分配唯一id|静默
|3|调用api上传所有媒体文件，并返回一个本地-云端媒体的对应关系|静默
|4|更改markdown裸文件，更新媒体文件的引用|文本编辑器更新
|5|调用api发布文章，然后遵循云端发布的逻辑（步骤3及以后）|地址栏url变为`editor?id=[步骤2分配的id]`



## url query处理
**以下逻辑仅在已登录时生效，在未登录时会直接先跳转登录，登录后再回调**
1. 只有`id=new`，则向服务器请求分配id，若失败，则回退到本地分配，然后清空编辑区，创建一篇新的云端文章，路由变为`id=new-[allocated id]`（已解决）
2. `id=new-xxx`，请求api验证可用性（验证id格式是否正确，有没有冲突），如果命中已有id，则调取已有的文章进入编辑模式，地址栏变为`id=xxx`（去掉`new-`），如果格式正确且未命中已有id，则视作新文章，不做额外处理；如果都不满足前述条件，则视作`id=new`处理（已解决）
3. `id=xxx`，向服务器请求是否有对应id的文章，若有，则进入云端编辑模式，加载云端文章内容；否则回退至`id=new`（已解决）
4. 其他非法请求，回退至本地editor，(no query)（未解决）

*`xxx`表示任意长度的任意字符串*


## POST /api/post
无论是否存在现有文章，都必须提供id和是否新文章（newPost: boolean）。
1. 先验证id格式是否符合，不符合则400；
2. 先根据`newPost`判断：
 - 标记为新文章`newPost === true`：id命中一个已存在的文章，则返回400，若未命中，则使用这个id创建新文章；
 - 未标记为新文章`else`：id命中一个已存在的文章，则编辑现有文章，若未命中，则使用这个id创建新文章；
3. 不提供id的所有请求一律400。
**编辑器对接：**
- 在触发publish或者save draft后，有以下case：
|case|请求id|newPost值|api返回|编辑器行为|
|---|---|---|---|
|新文章|合法新id|true|200（保存成功）|地址栏由 `editor?id=new-[valid-new-id]` 变为 `editor?id=[valid-new-id]`
|新文章|已有id|true|400（已有文章）|请求服务器分配新id，地址栏由 `editor?id=new-[exist-id]` 变为 `editor?id=new-[new-alloc-id]`，编辑器内容不刷新，仅修改“当前编辑文章的id”，并在客户端提示使用新id重试
|新文章|非法id（*）|true|400（非法id）|请求服务器分配新id，地址栏由 `editor?id=new-[illegal-id]` 变为 `editor?id=new-[new-alloc-id]`，编辑器内容不刷新，仅修改“当前编辑文章的id”，并在客户端提示使用新id重试
|编辑文章|已有id|false|200（保存成功）|地址栏保持 `editor?id=[valid-new-id]`
|编辑文章|合法新id (*)|false|200（保存成功）|地址栏保持 `editor?id=[valid-new-id]`
|编辑文章|非法id（*）|false|400（非法id）|提示用户id非法，请重新打开编辑器，地址栏保持 `editor?id=[illegal-id]`，编辑器内容不刷新

(*)标记的情况在正常交互下不会出现

|---|---|---|
