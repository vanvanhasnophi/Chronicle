# 主题与插件分层规范

版本：1.0

适用范围：`astro-template`

目的：在保留 Astro `src/pages` 路由语义的前提下，建立“公共能力 + 主题包 + 插件系统”的分层约定，使社区可以安全地改组件样式、页面布局、以及新增功能页面，同时避免业务逻辑散落在页面层。

---

## 新原则：轻定制化优先

为适配本仓库以单人维护为主、面向少数开发者的目标，本项目采用“轻定制化优先”原则：

- **保持内聚与可修改性**：优先保持模板和主题实现的内聚、减少多层抽象，方便维护者快速理解与修改；
- **明确定制点**：通过主题变量、少量可替换组件与清晰的配置项来支持样式与行为定制；
- **避免过度解耦**：仅在确有长期收益或复用需求时引入额外抽象；不要为适配“完全零配置的外部使用者”而牺牲可读性；
- **对开发者友好，不针对初学者**：文档与接口应服务于具备基础前端经验的开发者，而非完全的入门用户。

在后续改造中，可以接受页面保留适度的实现细节以降低重构成本，但重大或复杂的共享逻辑仍应放入 `utils`/`shared`，并通过注释说明设计取舍。

---

## 1. 基本原则

1. `pages` 负责路由入口，不负责数据处理实现。
2. `layout`、`components`、`styles` 负责主题表现，不负责核心业务规则。
3. `utils`、`shared` 负责可复用的数据处理逻辑，优先保持纯函数。
4. `core` 负责站点级基础能力和稳定协议，不依赖主题实现。
5. `plugins` 负责可插拔功能扩展，只能通过约定的扩展点接入。
6. 主题和插件都可以扩展站点能力，但不得直接修改公共核心契约。

---

## 2. 目录规划

建议目录如下：

```text
src/
  core/          # 框架无关的站点核心能力
  shared/        # 通用业务逻辑与可复用能力
  utils/         # 纯工具函数与无状态处理
  contracts/     # 主题协议、插件协议、共享类型
  plugins/       # 社区插件与插件注册入口
  theme/         # 当前主题的专属实现
  layouts/       # 主题布局层
  components/    # 主题组件层
  styles/        # 主题样式层
  pages/         # 路由入口，只保留薄层
  middleware.ts  # 运行时适配与请求处理
```

### 2.1 目录职责

- `core`：站点配置、API 连接规则、资源基地址、内容模型、扩展协议的基础定义。
- `shared`：跨主题、跨插件复用的业务能力，例如内容转换、站点配置归一化。
- `utils`：纯函数工具，例如 Markdown 解析、日期格式化、标签处理、路由辅助。
- `contracts`：类型定义和接口约定，包含主题协议、插件协议、站点上下文类型。
- `theme`：主题特有的页面片段、组件组合、视觉变量和主题默认行为。
- `plugins`：社区扩展实现，只通过协议挂接，不直接侵入页面内部。
- `layouts`、`components`、`styles`：当前主题的实现层，允许被主题包替换。
- `pages`：Astro 路由入口，页面文件只做数据装配和渲染调用。

---

## 3. 依赖方向

依赖关系必须保持单向：

```text
pages -> theme -> shared -> core
pages -> utils
plugins -> contracts -> core
```

### 3.1 禁止的依赖

- `core` 不得依赖 `theme` 或 `plugins`。
- `utils` 不得依赖页面组件或 Astro 页面上下文。
- `plugins` 不得直接修改主题源文件。
- `pages` 不得实现 Markdown 解析、TOC 生成、HTML 清洗、标签映射等数据处理逻辑。

---

## 4. 页面约束

`src/pages` 是路由入口层，只允许做以下事情：

1. 读取内容数据。
2. 调用 `utils` 或 `shared` 完成数据处理。
3. 将数据传入 `layout`、`theme` 或页面级组件。
4. 输出页面结构。

### 4.1 页面中禁止出现的实现

以下逻辑必须移出 `pages`：

- Markdown 解析和内容清洗。
- TOC 构建和目录提取。
- KaTeX 预处理和数学公式替换。
- 标签、日期、路由等规则化处理。
- API 地址拼接和环境差异判断。
- 站点级 feature flag 的归一化逻辑。

### 4.2 页面新增规则

当社区新增页面时：

- 若是站点公开路由页面，放入 `src/pages`。
- 若是主题内部页面片段或可复用区块，放入 `src/theme`。
- 若是插件提供的新增功能页，必须通过插件协议注册路由，而不是直接改公共核心文件。

---

## 5. 主题约定

主题只负责“看起来是什么样”和“页面怎么组织”。

### 5.1 允许修改的内容

- 页面布局结构。
- 组件样式与交互。
- 主题级变量、配色、字体、背景。
- 主题级导航、首页、文章页、列表页样式。
- 页面级展示逻辑的组合方式。

### 5.2 不允许修改的内容

- 公共数据模型定义。
- 核心 API 协议。
- 插件注册协议。
- 内容解析算法的基础约定。

### 5.3 主题包标准

主题包必须至少提供：

- `layouts/`
- `components/`
- `styles/`
- `pages/` 的薄层入口或页面片段
- 主题说明文档

---

## 6. 公共代码约定

### 6.1 `core`

`core` 存放站点稳定能力，优先保持低耦合、低变动。

建议包含：
- 站点运行时设置标准化。
- API 连接规则。
- 媒体基础地址处理。
- 站点上下文。
- 主题/插件协议基础类型。

### 6.2 `shared`

`shared` 存放可复用业务能力，允许比 `utils` 更有业务语义，但仍应跨主题可用。

建议包含：
- 内容模型转换。
- 文章元数据归一化。
- 导航数据组装。
- 站点配置归一化。

### 6.3 `utils`

`utils` 只放纯函数和无状态处理。

建议包含：
- Markdown 解析。
- TOC 构建。
- 日期格式化。
- 路由拼接。
- 标签处理。
- 图标映射。
- 主题变量归一化。

---

## 7. 插件协议

插件是面向社区的扩展单元，目标是“新增能力，不破坏主题”。

### 7.1 插件必须声明的信息

每个插件至少应声明：

- 名称。
- 版本。
- 作者。
- 依赖的核心版本。
- 支持的扩展点。
- 是否涉及新增路由。
- 是否依赖特定主题。

### 7.2 插件允许做的事情

- 增加新页面。
- 增加侧栏、页脚、文章附加区块。
- 增加短代码或渲染前后处理。
- 增加导航项或菜单项。
- 读取站点上下文并生成派生内容。

### 7.3 插件不允许做的事情

- 直接改写 `core` 文件。
- 直接修改主题内部私有实现。
- 绕过注册协议注入全局副作用。
- 将数据处理逻辑写回 `pages`。

### 7.4 推荐扩展点

建议预留以下扩展点：

- `onSiteInit`
- `onContentLoaded`
- `onPageContext`
- `onNavBuild`
- `onBeforeRender`
- `onAfterRender`
- `extendRoutes`
- `extendMenu`

---

## 8. 版本与兼容

1. 主题包与插件包都应声明支持的核心版本。
2. 核心协议变化必须先升级 `contracts`，再调整主题和插件。
3. 插件优先使用新增字段，不得依赖未声明的内部实现。
4. 若协议升级会破坏旧插件，应提供兼容层或迁移说明。

---

## 9. 迁移步骤

建议按以下顺序改造现有项目：

1. 先把 Markdown 解析、TOC、标签、日期等逻辑从 `pages` 移到 `utils` / `shared`。
2. 再把页面结构提炼到 `theme`，保留 `pages` 作为薄层。
3. 将运行时配置和 API 地址归一化到 `core`。
4. 最后补充 `contracts` 和插件注册机制。
5. 每次迁移后运行构建，确认路由与页面输出未变化。

---

## 10. 验收标准

满足以下条件即认为规范可执行：

- `pages` 中不再出现内容解析实现。
- 主题可以替换，但路由不变。
- 新功能页面可通过插件接入，而不修改核心代码。
- 公共逻辑可被多个主题复用。
- 构建通过，主路由和文章路由输出正常。

---

## 11. 当前仓库的落点建议

基于当前仓库结构，建议优先落地以下改动：

- 保留 `src/pages` 路由结构不变。
- 把 `markdownParser.ts`、`toc.ts`、`dateUtils.ts`、`tagUtils.ts` 继续作为公共逻辑候选。
- 把 `Layout.astro`、`NavHeader.astro`、`styles/` 视为主题层。
- 新功能页面优先通过 `src/pages` 作为路由入口，再由 `theme` 或插件提供展示和扩展。
- 主题特有逻辑与公共逻辑严格分目录。

---

## 12. 对齐主流静态网站生成器架构

为了保持与主流静态网站生成器的一致性，本项目建议采用“内容层、展示层、扩展层、适配层”四层模型。这样既符合 Astro、Hugo、Docusaurus、Nuxt static、Gatsby 等常见 SSG 的组织方式，也便于未来替换主题或接入插件而不破坏路由与内容结构。

### 12.1 四层模型

- 内容层：负责文章、页面、集合、导航、站点配置等内容源。
- 展示层：负责布局、页面、组件、样式、主题变量。
- 扩展层：负责插件、短代码、新页面能力、增强渲染钩子。
- 适配层：负责框架相关的构建入口、路由接入、环境变量、SSR/SSG 差异处理。

### 12.2 与主流 SSG 的对应关系

| 通用 SSG 概念 | 本项目对应层 | 说明 |
| --- | --- | --- |
| Content / Content Collections | `core` / `shared` / `utils` | 统一内容模型、页面元数据和转换逻辑。 |
| Layouts / Themes | `layouts` / `theme` / `styles` / `components` | 负责页面视觉和结构组合。 |
| Pages / Routes | `src/pages` | 只保留路由入口和薄层装配。 |
| Plugins / Integrations | `plugins` / `contracts` | 通过协议注入新功能，不直接侵入主题实现。 |
| Adapters / Output Targets | `astro.config.mjs` / `middleware.ts` / 构建脚本 | 对接静态输出、开发服务器与部署环境。 |
| Data / Derived Data | `shared` / `utils` / `core` | 站点级派生内容与规则化处理。 |

### 12.3 主流 SSG 对齐原则

1. 路由优先由文件系统决定，页面只负责呈现，不承担核心业务处理。
2. 主题应视为可替换包，页面结构和组件组合可变，但内容模型应稳定。
3. 插件应通过显式注册和钩子工作，而不是通过修改页面内部实现工作。
4. 适配层应集中处理框架差异，例如静态输出、代理、预渲染、客户端注入。
5. 公共逻辑应尽量无框架依赖，避免锁死在某一 SSG 的实现细节中。

### 12.4 目录演进建议

若后续继续向主流 SSG 架构靠拢，建议逐步引入以下概念目录：

```text
src/
  content/    # 内容源或内容索引的统一入口
  data/       # 结构化站点数据
  theme/      # 可替换主题包
  plugins/    # 社区扩展
  adapters/   # 框架或部署适配器
```

其中：

- `content/` 更接近主流 SSG 的内容入口概念。
- `theme/` 对应主题包或皮肤包。
- `plugins/` 对应社区扩展。
- `adapters/` 对应输出目标和运行时适配。

### 12.5 落地判断标准

如果一个改动符合以下条件，通常就符合主流 SSG 的架构习惯：

- 它不会要求页面层直接实现数据处理。
- 它可以在不改核心内容模型的前提下替换主题。
- 它可以通过插件或适配器新增能力，而不是硬编码在页面里。
- 它的构建输出仍然是稳定的静态站点产物。

---

备注：本规范面向 `astro-template` 的社区主题化与插件化演进，后续可继续补充插件 manifest 示例和主题包示例。
